# Key Management

## Runtime keys

| Key | Env var | Format | Use |
|---|---|---|---|
| JWT signing secret | `JWT_SECRET` | At least 32 random bytes, stored as text | HMAC JWT access tokens |
| PHI master key | `MEDSEC_MASTER_KEY` | Base64-encoded 32 bytes | HKDF source for field encryption and CNP HMAC |
| Evidence signing key | `EVIDENCE_ED25519_PRIVATE_KEY` | Base64 Ed25519 private key (64 bytes) or seed (32 bytes) | Sign evidence-chain hashes |
| MQTT CA key | `secrets/ca.key` | PEM | Sign local broker/client certificates |

Generate development values:

```bash
scripts/gen_keys.sh
```

Generate local mTLS certificates:

```bash
scripts/gen_certs.sh
```

The Ed25519 value generated by `scripts/gen_keys.sh` is a 32-byte seed. The API expands it to the 64-byte private key at startup.

## Rotation

1. Stop ingestion and report exports.
2. Generate a new key.
3. For `JWT_SECRET`, deploy and force login refresh.
4. For `MEDSEC_MASTER_KEY`, decrypt existing encrypted fields with the old key and re-encrypt with the new key in a controlled maintenance script.
5. For `EVIDENCE_ED25519_PRIVATE_KEY`, start a new chain segment or preserve the old public key with archived chain metadata.
6. For mTLS certificates, regenerate server/client certs and restart Mosquitto plus clients.

## Repository hygiene

`.env` and `secrets/*` are now ignored for future changes. Because this repository already tracks local demo secrets, rotate all committed certificates and credentials before sharing the repository. Use `git rm --cached` during a reviewed cleanup commit so local files can remain on disk while leaving version control.
